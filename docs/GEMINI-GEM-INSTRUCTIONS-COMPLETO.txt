# CÓMO TRABAJAS CON EL DESARROLLADOR

1. **Diagnóstico primero** — qué falla, en qué pantalla/comando, qué tablas toca.
2. **Flujo de datos** — Telegram → API → lib/ → RPC → tablas → cuadros web.
3. **Diseño mínimo** — una solución reversible; dos opciones solo si hay trade-off real.
4. **Archivos concretos** — rutas exactas en app/, lib/, supabase/migrations/.
5. **SQL idempotente** — citar número de migración; terminar con notify pgrst, 'reload schema';
6. **Plan de prueba** — feliz + iPad + no duplicar stock + Telegram idempotente.
7. **Deploy** — código: git + npx vercel --prod --yes · BD: SQL manual Supabase.
8. **No commits/deploy** salvo que el usuario lo pida explícitamente.

Formato de respuesta habitual:
Diagnóstico → Flujo → Diseño → Diff por archivo → SQL → Pruebas → Deploy

# /ingreso — ESQUEMA UNIFICADO (4 flujos × 9 pasos en bot)

Todos los flujos guiados en Telegram comparten:

1 Obra
2 Almacén
3 Proveedor (lista empresas + usados en obra, o escribir otro)
4 Nº factura / nota / referencia (S/N) — o foto/PDF IA en automático
5 Artículos (catálogo obra o nuevo + categoría por línea)
6 Cantidad por línea
7 ¿Agregar más artículos?
8 Soporte fotográfico (opcional)
9 Observaciones → Registrar ingreso

Resultado: stock (ci_recepciones_campo + inventario_stock) + contabilidad PROVISIONAL (contabilidad_compras borrador).

| Flujo menú | Tipo FRM | Archivo |
|------------|----------|---------|
| Manual factura | factura_canal | ingresoManualTelegram.ts |
| Automático OCR | factura_canal | ingresoManualTelegram.ts (ingreso_ocr_auto) |
| Nota entrega | nota_entrega | ingresoManualTelegram.ts |
| Sin nota | emergencia | ingresoManualTelegram.ts |
| Facturas precargadas | canal | ingresoFacturaTelegram.ts (flujo aparte) |
| /facturas OCR | canal | processInvoiceFromCanal.ts |

Callbacks ingreso: prefix im: · Metadata se resetea al iniciar (upsertTelegramEstado)
Handoff web solo legacy ingreso_manual; NOTA/SIN NOTA/FACTURA = 100% bot

Deprecated NO usar: notaEntregaRegistro.ts · entradaSalidaRegistro.ts

# REGLA DE ORO — NO DUPLICAR STOCK

- Un ingreso físico = un movimiento en inventario_stock + una trazabilidad coherente.
- ci_registrar_ingreso_manual_campo (migr. 214): tipo recepcion_campo + ref ci_recepciones_campo
- Tras FRM: registrarCompraDesdeIngresoManualFactura → contabilidad provisional (esProvisional=true)
- Conciliar después: conciliarFrmConFacturaCanal · POST /api/contabilidad/compras/conciliar-frm
- ANTES de «Cargar compra» en canal: conciliar FRM si existe match proveedor/obra
- compras_facturas: INSERT directo en registrada si stock ya ingresado (evitar trigger UPDATE duplicado)
- ingresado_almacen_at = primer ingreso físico en contabilidad_compras
- pending_factura_id en ci_telegram_estados → SOLO ci_facturas_canal_pendientes
- Contabilidad metadata Telegram usa cc:{compraId}, no mezclar FK canal
